# Development Rules

## 1. 总原则
- 所有任务先产出计划，再改代码
- 优先复用现有 1.0 能力
- 每次改动必须说明影响目录
- 每个新模块必须有 README
- 接口改动必须同步更新 API 文档
- 重大架构调整必须写入 `08-decisions/`

## 2. 输出规范
默认输出以下内容：
1. 任务目标
2. 涉及目录
3. 拟新增/修改文件
4. 风险点
5. 验收标准
6. 下一步建议

## 3. 代码规范
- 命名清晰、避免缩写歧义
- 新增服务优先模块化，不做无意义过度微服务
- 领域对象命名以业务含义为准
- 禁止跨模块硬编码依赖
- 配置与密钥不得写入仓库

## 4. AI 代理执行边界
- OpenClaw：总控、拆解、汇总、协同
- Codex：标准开发、骨架生成、测试补充
- Claude Code：复杂重构、旧系统适配、深度审查
- Antigravity：实验与多代理可视化，不作为唯一事实源

## 5. DoD（Definition of Done）
一个任务完成至少满足：
- 需求说明完整
- 代码或文档已提交
- 风险点已列出
- 验收标准已对齐
- 影响范围清楚
- 如有接口变更，文档已更新
